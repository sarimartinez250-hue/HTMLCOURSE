ALTERNATIVE TEXT (Texto Alternativo)

Es el texto que acompaña a la imagen describiendo su contenido para que en caso de que la URL de la imagen sea incorrecta, la imagen no se haya descargado aún o tiene un formato no soportado el usuario vea esa descripción y pueda hacerse una imagen mental sin necesidad de ver la imagen.

También sirve para accesibilidad, para hacer nuestras páginas inclusivas en caso de que personas ciegas que usen screenreader y para SEO (accesibilidad de Google) ya que Google no sabe que hay ahi, es un atributo obligatorio después de src.

Hay ocasiones donde alt se coloca vacio o se puede omitir cuando la imagen es meramente decorativa y su contenido no brinda mayorr utilidad al codigo.

Las descripciones deben ser muy especificas de acuerdo a la imagen, sin extendernos en descripcion cosas sencillas pero que describan a la perfeccion.

    CROSSORIGIN

    Este atributo que no solo se usa en img, tiene la funcion de indicar si la busqueda del elemento en este caso la imagen debe hacerse por CORS (Cross-Origin Resourse Sharing) o no. En otras palabras, le dice al navegador si puede utilizar ese recurso externo sin generar problemas de seguridad o restricciones cuando el usuario interactue con él, especialmente cuando se usa la etiqueta de canvas (que se vera mas adelante) para editarlo, dibujar sobre él o descargarlo. Tiene dos valores principales:

    • "anonymus":Indica que el recurso puede cargarse sin enviar credenciales del usuario (como cookies, tokens o sesiones). Esto es lo mas seguro y lo mas usado. Permite que la imagen siga siendo válida para manipularla en un elemento canvas sin contaminarlo. Si la imagen externa permite CORS ya que es necesario aclarar que no todos los sitios lo permiten, pero si lo hace el navegador la cargara correctamente y el canvas funcionara sin restricciones.
    • "use-credentials":Esta propiedad envia credenciales del usuario al servidor externo. Esto casi nunca se usa para las imagenes porque implicaria compartir cookies o sesiones con otro dominio, lo cual puede ser inseguro. Solo se usa cuando el servidor externo lo necesita especificamente lo cual es raro en recursos publicos..
    Si no se pone crossorigin y mas que todo con "anonymus" como propiedad nos exponemos o como programadores, exponemos al usuario a dos situaciones:
    • Situacion 1: Aca la imagen se muestra normalmente, pero no se puede usar en canvas ya que el canvas queda contaminado (tainted) y expone a situaciones de seguridad si se usa aun asi.
    • SItuacion 2: En este caso el navegador puede bloquear el recurso si el servidor tiene restricciones. Entonces el archivo no se muestra.
    Y como aclaramos, cross-origin no brinda los permisos, quien los brinda es el servidor externo de donde proviene el recurso que estamos manipulando, entonces en casos que esos servidores no lo admitan aunque tengamos cross-origin no vamos a poder manipular el elemento en un canvas.

    HEIGHT

    Es la altura de la imagen en pixeles con la que se mostrara en la pagina web, este atributo le indica al navegador con que altura exacta debe mostrarla. En HTML5 es un valor que unicamente se expresa con numeros, el navegador interpreta que son pixeles, no se puede poner height=50%, es incorrecto. En HTML4 height si aceptaba otro valor que era porcentaje, donde este valor era decirle al navegador el tamaño que usaria respecto al contenedor donde esta se encontrara, por ejemplo si se colocaba un valor de height de 25%, el navegador mostraria la imagen con un tamaño de la cuarta parte del valor total de la altura del contenedor. Hoy en dia, lo profesional es hacerlo mediante CSS, directamente en HTML con un style o mediante una clase desde CSS. Si solo se coloca height sin modificar width (ancho), el navegador lo mantiene proporcional, pero si manipulamos ambos de una forma desproporcional, la imagen podria quedar deformada, alargada o aplastada.

    REFERRERPOLICY

    Política de referencia o referrerpolicy es una cadena que indica que información mostrar al servidor externo donde se encuentra alojada la imagen que está siendo solicitada, el navegador envia un mensaje llamado referer el cual al ser abierto por el servidor puede encontrar información del origen desde esta siendo solicitando dicha imagen. Un origen se compone de 3 elementos: protoloco o esquema, dominio o anfitrión (host) y puerto y la URL completa esta formada por esos 3 elementos y la ruta (route) o camino (path). Un downgrade ocurre cuando un recurso es solicitado desde un esquema HTTPS a un HTTP, ya que es un esquema HTTP no cifra su información y un intercambio entre protocolos HTTPS -> HTTP es como gritar la información a todo el mundo. Un localhost es tratado por los navegadores como un entorno de desarrollo confiable. Al solicitar un recurso HTTPS desde un local host, no se considera downgrade, por lo que el servidor puede recibir el origen según la politica aplicada.

    Referer
    Al ser abierto por el servidor recibe algo como:
    https + midominio.com + 443
    Los valores del atributo referrerpolicy son:

    • "no-referrer": En este caso el servidor recibe un referer que se encuentra vacío, no se le envía ningún tipo de información.

      Ejemplo: 
                                  
                                <img src="https://www.google.com/babobab.jpeg"
                                referrerpolicy="no-referrer"
                                alt="Árbol de Baobab">
      El servidor no recibe ninguna información.

    • "no-referrer-when-downgrade": Fue el comportamiento por defecto por muchos años. Se envía el origen (protocolo, dominio y, puerto) siempre y cuando no haya downgrade, si es de https a http no se envía nada.

      Ejemplo: 
                                      
                                    <img src="http://google.com/baobab.jpeg"
                                    referrerpolicy="no-referrer-when-downgrade"
                                    alt="Árbol de Baobab">
      Como el origen desde donde se esta solicitando es un https pero donde se encuentra la imagen tiene un esquema http, el servidor no recibe ninguna información.

    • "origin": Se envía el origen pero no la ruta.

      Ejemplo: 
                                      
                                    <img src="https://google.com/baobab.jpeg"
                                    referrerpolicy="origin"
                                    alt="Árbol de Baobab">
      El servidor recibe un referer que contiene el origen pero nunca la ruta.

    • "origin-when-cross-origin":Si es el mismo origen se envia la URL completa (protocolo+dominio+puerto+ruta), si es un origen distinto solo envía el origen.

      Ejemplo: 
                                      
                                    <img src="https://google.com/baobab.jpeg"
                                    referrerpolicy="origin-when-cross-origin"
                                    alt="Árbol de Baobab">
      Como la imagen esta siendo solicitada desde un localhost hacia un https, el servidor envia un referer que contiene unicamente el origen pero no la ruta.

    • "strict-origin": Envía solo el origen siempre y cuando no haya downgrade. Pero la ruta nunca es enviada asi pertenezcan al mismo origen.

      Ejemplo: 
                                      
                                    <img src="https://google.com/baobab.jpeg"
                                    referrerpolicy="strict-origin"
                                    alt="Árbol de Baobab">
      No hay downgrade por lo tanto google.com puede recibir el origen sin problemas.

    • "strict-origin-when-cross-origin":Si la solicitud va al mismo origen se envia la URL completa, si es hacia otro origen solo se envia el origen.

      Ejemplo: 
                                      
                                    <img src="google.com/babobab.jpeg"
                                    referrerpolicy="strict-origin-when-cross-origin"
                                    alt="Árbol de Baobab">
      Aquí solo se envía el origen.

    • "unsafe-url": Esta es una práctica muy riesgosa e insegura ya que el referer envía la URL completa al servidor habiendo o no downgrade lo cual puede generar una fuga de los origenes del recurso protegido por TLS hacia origenes inseguros.

      Ejemplo: 
                                      
                                    <img src="https://google.com/baobab.jpeg"
                                    referrerpolicy="unsafe-url"
                                    alt="Árbol de Baobab">
      La practica más insegura, aunque no hay downgrade en este caso, siempre es enviada la URL completa del navegador hacia el servidor. Incluso aunque el recurso este protegido por HTTPS, la URL completa puede filtrarse a servidores inseguros.
    NOTA: Actualmente, el valor por defecto en la mayoría de navegadores modernos es strict-origin-when-cross-origin En conclusión, referrerpolicy es un atributo que ayuda a mantener la seguridad cuando un recurso externo es solicitado, evita la filtración de información sensible sobre el origen desde donde se solicitan los recursos.

    SIZES

    El atributo sizes es una lista de una o más cadenas separadas por coma, que le indica al navegador cuánto espacio ocupará la imagen en el diseño, dependiendo el ancho del viewport. Cada tamaño de la fuente consiste en:

    1. Condición de medios. Se omite en el ultimo item (es decir, no se agrega una condición para el último medio).
    2. Valor del tamaño. Especifica el tamaño y el ancho CSS que ocupará la imagen en el layout (px, vw, em, etc.)

    Este atributo se utiliza juntamente con el siguiente atributo llamado srcset.

    El atributo sizes solo tiene efecto cuando el atributo srcset utiliza descriptores de ancho (w). Sin srcset, sizes es ignorado por el navegador. Cuando se usa descriptores de densidad (x), sizes no participa en la selección.

    Este atributo funciona con dos tipos de condiciones: max-width y min-width.

    1. max-width (Mobile First): Significa del más pequeño hacia el más grande, dándole prioridad a los dispositivos móviles. Es el más usado, porque coincide con diseño responsivo moderno, es más fácil de razonar y se alinea con CSS mobile-first.
    2. min-width (Desktop First): Escritorio primero. Del más grande al más pequeño, es menos común porque al priorizar primero el escritorio requiere más cuidado para evitar errores de diseño en pantallas pequeñas.
    Sintaxis básica: 
                            
                            sizes="
                            (max-width: 600px) 100vw,
                            (max-width: 900px) 50vw,
                            33vw"
    Se lee:
    1. En viewports menores o iguales a 600px la imagen ocupara el 100% del ancho visible.
    2. Si el viewport mide hasta 900px, la imagen ocupará el 50% del ancho visible.
    3. En pantallas más grandes que no se adaptan a las dos condiciones anteriores ocupará el 33% del ancho visible (una tercera parte)
    vw significa viewport width (ancho de la pantalla visible).

    SRCSET

    El atributo srcset es una lista de una o más cadenas separadas por coma indicando las posibles fuentes para usar. Cada cadena esta compuesta por:

    1. URL de la imagen.
    2. Opcionalmente, espacios en blanco seguidos de:
      • Un ancho, que es un entero positivo seguido directamente por "w".
      • Densidad del pixel, un positivo decimal seguido directamente de "x".

    Si no especificamos descriptores, la fuente es asignada por defecto a densidad de 1x.

    No se deben mezclar descriptores de ancho con descriptores de densidad, porque son dos modelos totalmente distintos que no se complementan. Cuando srcset utiliza descriptores de ancho (w), el atributo src actúa como fallback para navegadores que no soportan srcset. Y cuando utiliza un descriptor de densidad, src actúa como un recurso 1x real.

    Los espacios entre los caracteres de la cadena son opcionales ya que son omitidos por el navegador tanto los del inicio como del final, pero considerando el ejemplo de que en la primera cadena de caracteres no haya espacios ni descriptores y si se tiene una segunda cadena, esta segunda cadena debe iniciar uno o más espacios después de la coma si no la coma se toma como parte de la cadena.

    Los agentes del usuario aplican algoritmos internos que consideran DPR (Device Pixel Ratio), ancho de renderizado, preferencias del usuario y condiciones de red al momento de elegir la imagen. Esto quiere decir que no se basan solo en la obtención de la resolución si no también en:

    • Preferencias del usuario:
      • Ahorro de datos activo o no.
      • Modo de bajo consumo.
      • Uso de accesibilidad o zoom.
    • Condiciones de red: El navegador puede evaluar:
      • Red lenta (2G/3G), red media (4G) o red rápida (5G).
      • Alta latencia.
      • Conexión inestable.
    • Ahorro de ancho de banda: El navegador prefiere una imagen "suficientemente buena" antes que una perfecta pero pesada.
    • Heurísticas internas (decisión inteligente): O sea no siempre eligen la imagen "ideal", eligen la mejor decisión contextual para el dispositivo.

                    <img 
                    src="easterbunny.png"
                    alt="Conejo de pascua"
                    srcset="
                    easterbunny-400.png 400w,
                    easterbunny-800.png 800w,
                    easterbunny-1200.png 1200w
                    ">
    • Debe existir cada imagen fisicamente con ese tamaño de ancho.
    • Los valores 400w, 800w y 1200w indican el ancho real de la imagen, no el tamaño con el que se mostrará.
    • El navegador no descarga todas las imágenes, solo la que considere más adecuada.

    ¿Cómo se complementan sizes y srcset?

    Estos atributos juntos permiten al navegador elegir automáticamente la imagen más adecuada según el tamaño del dispositivo, la resolución de pantalla y el espacio real que ocupará la imagen en el diseño.

    Su objetivo principal es mejorar el rendimiento, ahorrar ancho de banda y mantener una buena calidad visual en distintos dispositivos (móviles, tabletas, computadoras, pantallas de alta densidad, etc.)

    Sintáxis que el navegador sigue al trabajar con estos atributos en conjunto:

    1. El navegador evalúa las condiciones declaradas en sizes.
    2. Calcula el espacio real que la imagen ocupará en pantalla.
    3. Considera la densidad de píxeles del dispositivo (DPR).
    4. Elige del srcset la imagen que mejor se adapte a ese espacio.
    5. Descarga solo esa imagen.
    Todo este proceso es realizado automáticamente por el navegador en milisegundos.

    NOTA: los descriptores no controlan el tamaño CSS con el que la imagen se muestra, sino la resolución de la imagen.

    Ejemplo usando sizes y srcset: 
                    
                    <img 
                    src="easterbunny.png"
                    alt="Conejo de pascua"

                    srcset="
                    easterbunny-400.png 400w,
                    easterbunny-800.png 800w,
                    easterbunny-1200.png 1200w"

                    sizes="
                    (max-width: 600px) 100vw,
                    (max-width: 900px) 50vw,
                    33vw"
    Se lee como:
    • Móvil: Mide 360px.
      1. El navegador evalúa las condiciones.
      2. Primera condición: Si la pantalla mide 600px o menos, la imagen se mostrará en el 100 del viewport. Esta primera condición cumple para el móvil.
      3. Ahora el navegador considera la densidad de pixeles del dispositivo, por ejemplo si la densidad es de 2x, el navegador multiplica el tamaño de la pantalla (360px) por el DPR (2x). 360 * 2 = 720.
      4. El navegador se va a srcset y elige la imagen que mejor se adapte.
        • 400w → Muy pequeña, se verá borrosa.
        • 800w → Perfecta para una densidad de 720px. ✓
        • 1200w → Demasiado alta, descarga innecesaria de recursos. ✗
      5. Descarga la imagen easterbunny-800.png para mostrarla en pantalla.
    • Tableta: Mide 800px. El navegador sigue la misma sintáxis anterior:
      1. Evalúa las condiciones.
      2. Primera condición no se cumple.
      3. Pasa a la segunda.
      4. Evalúa.
      5. Segunda condición: Si la pantalla mide hasta 900px, la imagen usará el 50% del viewport. 800 ≤ 900. Cumple ✓
      6. La evaluación se detiene ahí.
      7. La imagen usará el 50vw, o sea 400px en pantalla.
      8. Considera la densidad de pixeles del dispostivo, por ejemplo 2x, el navegador multiplica 400 * 2 = 800.
      9. El navegador se va a srcset a elegir la imagen:
        • 400w → Muy pequeña, la imagen se verá borrosa.
        • 800w → Perfecta ✓. No hay más ni menos.
        • 1200w → Demasiado alta. ✗
      10. El navegador descarga easterbunny-800.png para mostrarla.
    • Escritorio: Mide 1200px. Misma serie de pasos.
      1. El navegador se va a sizes a evaluar condiciones.
      2. Primera condición no cumple.
      3. Se va a la segunda.
      4. Segunda condición no cumple.
      5. Se va a la ultima cadena sin condición.
      6. Navegador interpreta como: Si 1200 no entra en 600px o menos ni 900px o menos, usaremos el 33% del tamaño del viewport. ✓
      7. Se queda con esta condición.
      8. La imagen ocupara la tercera parte del tamaño de la pantalla ≈ 396px.
      9. Considera la densidad de pixeles del escritorio 1x, realiza la multiplicación: 396 * 1 = 396 ≈ 400.
      10. Se va a srcset a elegir la mejor opción:
        • 400w → Perfecta. ✓
        • 800w → Podría se considerada por el navegador pero innecesaria.
        • 1200w → Demasiado, totalmente inútil. ✗
      11. El navegador descargará la imagen easterbunny-400.png para mostrar en la pantalla del escritorio.

    De esta forma es como sizes y srcset trabajan de la mano, no pueden ir la una sin la otra.

    Aclarando que todos esos pasos no son trabajo de nosotros como programadores, es parte del trabajo que realiza en milisegundos el navegador. Se muestran como un extra para comprender a fondo de una mejor manera al servidor.

    El uso de srcset o de sizes no omiten el uso de CSS para aplicar alto o ancho a la imagen, la imagen siempre los necesita!

    DECODING

    El atributo decoding le indica al navegador cuando y como debe decodificar la o las imágenes, es decir cuando convertir los bytes descargados en píxeles visibles para ser mostrados en pantalla.

    Empecemos por donde deberíamos, ¿qué es decodificar? Bueno, cuando nosotros descargamos una imagen y la cargamos en el editor de texto, al momento de hacer "running" a la página, el navegador no recibe el contenido de la imagen en si directamente, es decir, el navegador no ve formas ni colores pintados, recibe datos comprimidos en un archivo que puede ser jpg, png, jpeg, webP u otro; estos archivos contienen metadatos, comprensión, información del color y algoritmos matemáticos. Esto es lo que el navegador ve, o sea pixeles con información de la fila y columna y el color declarado en rgb, rgba, hexa, hsl u otra forma, entonces la decodificación es, convertir esos datos comprimidos en píxeles visibles y aquí es donde entra decoding.

    ¿Por qué? Porque la decodificación dependiendo el navegador puede ocurrir ya sea en el GPU (Unidad de Procesamiento Gráfico), en el CPU (Unidad Central de Porcesamiento), en el hilo principal o en hilos separados y aquí pueden haber problemas porque decodificar consume tiempo y recursos del CPU y la memoria y puede bloquar el flujo del código si se hace en el momento incorrecto, así como producir un retraso en la actualización lo que conocemos como "jank". So, aquí entra nuestro dichoso atributo de decoding y su valores.

    • "sync" (síncrono): Con este valor, el navegador decodifica la imagen sincrónicamente mientras se renderiza o ejecuta el resto del contenido del DOM y se procesa todo junto. En otras palabras, cuando el navegador llega a la parte del código donde la imagen se encuentra puede detener lo demás y realiza la decodificación de la imagen para presentarla junto con todo lo demás, aunque no siempre se producen bloqueos, pero la imagen es quien tiene prioridad y esto aunque no siempre, no puede ser tan visible o medible en gran cantidad puede detener el scroll en pantalla, parpadeos, quedarse en blanco y lags, es como si el navegador dijera me agarraste desprevenido, no tengo listo tu espacio, entonces en ese momento se pone a prepararlo junto con todo lo demás.
    • "async" (asíncrona): Es la más usada, aunque no por todos los navegadores, pero en el caso de Firefox, es su valor por default y es la decodificación de la imagen tras haber renderizado y presentado el resto del contenido del DOM. La decodificación de la imagen se realiza en segundo plano sin afectar el flujo de la página, sin bloquear render ni scroll. Si la imagen tiene un width y height eso reserva su espacio en pantalla, puede producir un ligero efecto de que la imagen aparece después.
    • "auto": Es el valor por defecto al no asignar ninguno de los anteriores valores. No se muestra preferencia por el modo de decodificación, aquí el navegador decide que es lo mejor para el usuario y este se basa en heurísticas considerando el rendimiento.

    En conclusión, decoding en la mayoría de los casos es una sugerencia al navegador de en que momento realizar la decodificación de la imagen, no es tanto una orden y no deberíamos preocuparnos tanto por él, solo es casos muy específicos como Juegos Canvas, Apps gráficas, usos especiales de JavaScript donde esto si puede generar una gran diferencia visible. Entonces, así, queda a criterio de cada uno. Y pues por eso, las imagenes de este manual quedan con el decoding "auto".

    ELEMENTTIMING

    El atributo elementtiming se usa para medir el tiempo de renderizado de un elemento específico, normalmente imágenes o elementos grandes que importa para la experiencia del usuario.

    Este atributo no cambia como se ve la imagen, cuándo ni cómo se descarga, solo sirve para observación y métricas. Es decir, elementtiming es como colocar un cronómetro invisible para medir la posible experiencia visual que el usuario tendría respecto a cierto elemento importante del contenido.

    Existe para responder preguntas como:

    • ¿Cuánto se tardó esta imagen importante en aparecer?
    • ¿Este banner se muestra rápido o lento?
    • ¿Este elemento clave está afectando el rendimiento percibido?
    Y se usa junto con:
    • Performance API
    • PerformanceObserver
    • Core Web Vitals
    Especialmente para medir:
    • Renderizado de elementos clave.
    • Y aunque no reemplaza LCP, permite identificar con precisión qué elemento fue responsable del LCP y cuánto tardó en mostrarse.

    ¿Qué es LCP? LCP (Largest Contentful Paint) es una métrica del navegador, que forma parte de Core Web Vitals que responde a la pregunta de: ¿Cuánto tiempo tarda en mostrarse el contenido principal de la página al usuario?

    Mide el tiempo en milisegundos (ms) desde que el inicio de la navegación hasta que el elemento visual más grande visible en el viewport inicial es renderizado. Este elemento suele ser una imagen grande, un video, un bloque de texto amplio o un contenedor tipo hero.

    Entonces... ¿qué hace elementtiming con respecto a LCP? Bueno, el navegador calcula automáticamente el LCP, nosotros no decidimos que elemento es LCP. Sin embargo con el atributo elementtiming nos permite medir exactamente cuándo apareció un elemento específico, lo que ayuda a comparar su tiempo con LCP o incluso a confirmar si ese elemento resultó siendo el LCP de la página.

    En otras palabras el LCCP no depende de cuál elemento pese más, tenga mayor resolución o aparezca primero en el HTML, sino de cuál ocupa mayor área visible dentro del viewport principal.

    Por ejemplo: El navegador comienza renderizando el contenido del DOM y se encuentra con un <h1> grande, que se convierte en candidato a LCP. Luego aparece una imagen que ocupa el 50% del ancho del viewport, convirtiéndose en nuevo candidato. Más adelante se renderiza un banner que ocupa el 80% del ancho visible, convirtiéndose en el candidato final. Si no aparece ningún elemento visual más grande, ese banner será el elemento LCP y su tiempo de renderizado será el valor reportado.

    En conclusión, elementtiming ayuda a diagnosticar y analizar LCP, pero no a calcularlo, ya que ese calculo es realizado por el navegador automáticamente.

    La medición se realiza desde una lista que se genera en JavaScript, por ejemplo: 

                        
                        JavaScript
                        const observer = new PerformanceObserver((list) =>{
                        for (cont entry of list.getEntries()){
                            console.log(entry.name);
                            console.log(entry.startTime);
                        }
                    });

                    observer.observe({type: element, buffered: true });
    Y al correr el codigo se generara una lista con: el nombre del elemento y el tiempo exacto en ms (milisegundos) de cuando la imagen fue presentada en el render.

    Pero, entonces, cuando y donde se usa? Aquí la respuesta a cada una.

      La mayoría de los proyectos NO lo necesitan. Se usa cuando:
    • Se trabaja con optimización de performance.
    • Se tienen métricas reales de usuario (RUM).
    • Se necesita medir LCP con precisión.
    • Se hacen dashboards de rendimiento.
    Se usa principalmente en los elementos: 
    • <img>
    • <video>
    • Elementos grandes del layout.
    Y sobre todo en:
    • hero images
    • banners
    • imágenes above-the-fold (arriba de todo).

    Entonces como conclusión, el atributo elementtiming mide el tiempo desde que el navegador empieza a renderizar el elemento hasta que es pintado y visible en pantalla, no desde antes, no después, si no justo cuando se carga para saber si se tardo 300ms o 2.5s. Así que su uso es opcional y esta orientado a métricas avanzadas.

    FETCHPRIORITY

    El atributo fetchpriority es un hint (pista) o más bien una sugerencia al navegador sobre la prioridad relativa que se debe usar al obtener una imagen. Es como decirle al navegador: "Ey, esta imagen tiene (o no) prioridad respecto a otras al momento de descargarse." No cambia absolutamente ningún aspecto visual, solo influye en el orden/prioridad de descarga. Sus valores son:

    • "high": Le sugiere al navegador que debería obtener con prioridad esa imagen respecto a las demás del DOM, se usa en imágenes que son visibles inmediatamente, son parte del contenido visual y probablemente sea el LCP del sitio.
      Normalmente suele ir acompañando a imágenes que van al principio, aunque también podrían estar en medio pero no al final, sino, no tendría sentido su uso.
      Solo debería o realmente debe existir solo una imagen declarada con fetchpriority="high" o en casos muy específicos dos, más de eso resultaría una mala práctica, generando confusión en el navegador y nosotros perdemos el beneficio.


    • "low": Sugiere obtener la imagen con una prioridad baja en comparación con otras imágenes.
      Se usa con imágenes decorativas, de galería, de artículos largos o imágenes que no afectan la percepción inicial.


    • auto: Aquí no se establece una preferencia para la prioridad de descarga, dandole la decisión final al navegador el cual usa heurísticas de:
      • posición
      • tamaño
      • visibilidad
      • tipo de recurso
      • red
      • CPU
      Este es el valor por defecto al no colocar ninguno de los dos valores anteriores o establecer un valor no válido.
    En conclusión, fetchpriority es una sugerencia al navegador sobre a cual imagen debería darle prioridad al momento de obtenerla, no obliga ni garantiza resultados, solo orienta al navegador y actúa como complemento de sus heurísticas internas.

    fetchpriority es una herramienta de optimización avanzada que debe usarse con criterio, especialmente para mejorar la percepción de carga ideal y métricas como LCP, sin reemplazar otros mecanismos como loading, sizes o srcset.

    HEIGHT

    El atributo height en la etiqueta <img> es la altura intrínseca de la imagen, en pixeles. Se coloca como un numéro entero sin unidad.

    En otras palabras, height sirve para que el navegador conozca la relación de aspecto (entre el ancho y la alutra de la imagen), ayudando así a reservar un espacio en pantalla a la imagen antes de que esta se descargue y pinte para evitar desplazamientos de contenido y el famoso CLS (Cumulative Layout Shift).

    Se escribe como un valor numérico sin unidad, este valor numérico siempre representa píxeles CSS pero no es CSS y no determina el tamaño visual final si CSS sobreescribe estos valores, es decir, el height declarado en HTML no declara que la imagen al pintarse en el sitio se verá de esa altura si en CSS se ha declarado otra altura, pero lo ideal sería que el valor de height y width declarados en HTML coincidan con el tamaño real del archivo para evitar romper la proporción, generar cálculos incorrectos en el navegador y no afectar a CLS.

    NOTA: Incluir la altura y ancho en una etiqueta de imagen en HTML permite que el navegador calcule la relación de aspecto (aspect ratio) de la imagen antes de cargarla.
    Esta relación de aspecto se utiliza para reservar el espacio necesario para mostrar la imagen, reduciendo o incluso evitando cambios de diseño al descargar la imagen y mostrarla en pantalla.
    Reducir los cambios de diseño es fundamental para una buena experiencia del usuario y un buen rendimiento web.

    WIDTH Y HEIGHT EN HTML

    Entonces como complemento, los atributos width y height declarados en un elemento <img> suelen generar confusión en primer lugar porque como mencionamos antes, podemos pensar que estamos declarando el tamaño visual final de la imagen en pantalla y que estos sustituyen su declaración en CSS, cuando no es así.

    Su función es reservar un espacio en pantalla para la imagen mucho antes de que esta sea pintada en el sitio, para así evitar efectos visuales como que el texto se mueve para que la imagen se muestre en pantalla.

    Ejemplo:
                    
                    <img
                    src="https://downloads.net/black-widow.jpeg"
                    alt="Araña viuda negra"
                    fetchpriority="high"

                    width="1280"
                    height="720"

                    sizes="(max-width:767px) 100vw,
                    80vw"

                    style="
                    width: 100%;
                    height: auto"

                    srcset="
                    https://downloads.net/black-widow-400.jpeg 400w,
                    https://downloads.net/black-widow-670.jpeg 670w,
                    https://downloads.net/black-widow-800.jpe 800w,
                    https://downloads.net/black-widow-1200.jpeg 1200w,
                    https://downloads.net/black-widow-1400.jpeg 1400w"
                    >
    ¿Qúe hace el navegador paso a paso:
    1. Lee el HTML.
    2. Llega la etiqueta <img> y ve: 
                              
                            <img width="1280" height="720">
    3. Calcula la relación de aspecto: 1280 / 720 = 16:9
    4. El navegador reserva un rectángulo de 16:9 en pantalla para la imagen, aunque la imagen aún no esté descargada.
      • Aquí no importa todavía el tamaño visual final de la imagen, solo su proporción.
      • No depende de vw.
      • Solo reserva proporción.
    5. Llega a sizes (aquí como vimos antes, se encarga del ancho que ocupará en el layout).
      • Realiza la evaluación según el tipo de dispositivo donde se encuentre.
      • Calcula DPR y se basa en otras heurísticas para elegir la resolución que mejor se adapte a esas condiciones.
    6. Se va a srcset y elige de las opciones la que mejor se adapte.
      7. Ahora viene el tamaño visual final.
    7. Se va al archivo CSS y ve: 
                              
                            CSS
                            img{
                                width: 100%;
                                height: auto;
                            }
      ¿Qué significan?
      • width: 100%; → La imagen ocupará el 100% del ancho disponible del viewport.
        Si el contenedor mide 960px, la imagen se renderiza visualemnte con un ancho de 960px, pero esa medida no proviene de los atributos de HTML sino del CSS.
      • height: auto; → La altura se adapta automáticamente.
        La proporción reservada sigue manteniendo, o sea 16:9.
    8. Ahora el navegador continúa renderizando el resto del contenido.
    9. Cuando la imagen ya esta descargada y lista para ser pintada:
      • Se pinta dentro del espacio ya reservado.
      • Nada salta.
      • Nada se mueve.
    Y así la imagen se adapta al contenedor pero el navegador ya sabía cuanto medía.

    En conclusión, width y height no indican cuánto medirá la imagen visualmente en pantalla, 16:9 no es una medida, es una proporción que describe que diseño tendrá la imagen: cuadrada, rectangular, como una historia de Instagram, un reel, etc.

    Los atributos width y height en HTML no definen el tamaño final, definen la proporción del elemento y permiten reservar espacio en el layout.
    El atributo sizes indica cuánto espacio ocupará la imagen según el viewport.
    srcset permite elegir la mejor resolución según el DPR y condiciones del usuario.
    CSS define el tamaño visual final manteniendo la proporción.

    LOADING

    El atributo loading en <img> es una "strong hint" (pista fuerte), no una orden implícita pero tampoco una pista débil, para el navegador que le indica cómo debería cargar la imagen. De forma inmediata o diferir la descarga hasta que la imagen se encuentre a una distancia considerable del viewport. Sus valores son:

    • "eager": Carga la imagen sin diferir su descarga, permitiendo que el navegador la obtenga tan pronto como lo considere oportuno durante el proceso de carga inicial.
      En otras palabras realiza la carga de la imagen de forma inmediata para ser presentada en pantalla, esto podría llevar a un consumo alto de ancho de banda y sobrecarga de recursos que pueden ralentizar la descarga de otro tipo de elementos, por eso no es recomendable tener más de una imagen declarada con loading="eager", en casos muy especifícos dos, porque al tener muchas compiten entre sí por el ancho de banda, pueden retrasar el LCP y saturan la cola de descargas.
      Es ideal para:
      • Imágenes encima de todo (above-the-fold).
      • hero images.
      • Posible candidato a LCP.
    • "lazy": Difiere o aplaza la carga de la imagen hasta que alcance una distancia calculada del viewport considerada por el navegador.
      El objetivo de loading="lazy" es evitar el consumo de ancho de banda de red (bandwidth) y almacenamiento necesario para procesar la imagen hasta que el navegador tenga una certeza razonable de que sea necesaria y asi mejorar el rendimiento del contenido en la mayoría de los casos.

      En otras palabras, el valor lazy le dice al navegador que no es una imagen crítica para el contenido y que se podría encontrar abajo del scroll y no es necesario gastar ancho de banda y red en cargarla de manera inmediata si no hasta que él crea que sera necesaria. Es ideal para:
      • Cuando la imagen aparece al hacer scroll.
      • Galería de imágenes o artículos largos.
      • No es crítica para la primera impresión.
      • Es decorativa o secundaria.


      Cuando se usa loading="lazy" es crítico reservar espacio con width y height (o aspect-ratio).
      ¿Por qué?
      • La imagen no se descarga inmediatamente pero el navegador sigue renderizando el resto del contenido y presentandolo, pero con un width y height el layout si se renderiza y reserva para la imagen cuando sea el momento de cargarla en cambio, si no hay espacio reservado para esa imagen cuando se cargue puede provocar salto visual o un CLS.
        • Sintáxis correcta: 
                                    
                                <img src="imagen-no-critica.jpeg"
                                loading="lazy"
                                width="1280"
                                height="720"
        La imagen no se carga inmediatamente pero con width y height el navegador si renderiza el layout y le reserva la proporción.
    Otro aspecto que vale la pena aclarar es que antiguamente, loading="lazy" funcionaba únicamente con JavaScript habilitado, si el agente del usuario permitía su uso con el script deshabilitado esto daba paso al servidor a realizar un "tracking", en español, un rastreo del scroll aproximado del usuario y el servidor podía rastrear cuántas imágenes y cuándo eran solicitadas.
    Esto hoy en la actualidad ya no funciona así, en el aspecto de que loading="lazy" es solo aplicabale con JS, ahora loading="lazy" es nativo del navegador, no depende de JavaScript y funciona incluso con JS deshabilitado.
    El uso de loading="lazy" no implica que el servidor rastree el desplazamiento del usuario.
    El navegador simplemente realiza la petición HTTP cuando decide que la imagen es necesaria y el servidor solo recibe solicitudes normales de recursos, sin información sobre scroll, posición o comportamiento del usuario.

    En resumen, loading es una indicación al navegador de cómo debería cargar la imagen: de manera inmediata si es parte del contenido principal, es una hero image o es posible LCP, no tener muchas (más de dos imágenes) declaradas con loading="eager" para evitar consumo excesivo de ancho de banda y de red.
    De forma diferida, cuando la imagen no es parte crítica del contenido, se encuentra abajo del scroll y es una imagen decorativa o es una galería de imágenes, al usar este valor loading="lazy", siempre es fundamental reservar un layout con un width y height para evitar saltos visuales y CLS.
    loading="lazy" funciona hoy en día aún sin JavaScript habilitado y no produce ningún tracking para el usuario por parte del servidor.

    USEMAP

    El atributo usemap es parte de una URL parcial (que empieza por #) de un mapa de imagen asociado a un elemento.

    Es decir, casi similar al atributo visto antes, ismap, el atributo usemap conecta una imagen con un mapa de áreas clicables definididas por los elementos <map> y por <area> . Es declarado en la etiqueta img empezando por un # precedido por el nombre del elemento (name) contenido en la etiqueta <map>.

    A diferencia del atributo ismap, donde el elemento <img> debe estar dentro de un elemento <a> para que sea válido; el elemento usemap no puede estar dentro de un elemento <a> ni <button>.

    Sintáxis: 

                    
                    <img src="mapaEEUU.png" 
                    usemap="#mapaEEUU" 
                    alt="Mapa de los Estados Unidos de América" />

                    <map
                    name="mapaEEUU" >
                    ...
                    </map>

    Con usemap concluimos el estudio de los atributos del elemento <img>. Profundizaremos en otro aspecto fundamental en la web moderna que involucra a este elemento.

    ACCESIBILIDAD

    Anteriormente, casi al principio de este documento vimos uno de los atributos más importantes de la etiqueta <img>, el atributo alt (alternative text o en español, texto alternativo), el cual describe el contenido de la imagen para los diferentes casos que se puedan presentar (la URL del archivo es inválida, la descarga no se pudo realizar o para lectores de pantalla), en esta ocasión, este apartado es sobre el último de esos casos, para accesibilidad.

    ¿Qué es accesibilidad? En términos dados por la W3C (World Wide Web), accesibilidad web significa que sitios web, herramientas y tecnologías están diseñadas y desarrolladas de tal manera que las personas con discapacidades puedan usarlas. La Accesibilidad Web abarca todas las discapacidades que afectan al acceso a la Web, como:

    • Auditivas
    • Cognitivas
    • Neurológicas
    • Físicas
    • Del habla
    • Visuales
    Pero no solo a ellos, sino también a las personas sin discapacidad de todas las edades, regiones y condiciones.

    En otras palabras, la accesibilidad web se trata de hacer de nuestros sitios un lugar donde personas con y sin discapacidad puedan navegar sin barreras.

    Entonces, ¿qué tiene que ver en este aspecto el atributo alt ? Si bien, como lo mencionamos al principio, este nos ayuda a describir el contenido de la imagen, si la imagen no se carga, la descripción contenida en el atributo alt será la que aparecerá en el lugar de la imagen, pero ¿para una persona que se auxilia de screen readers o lectores de pantalla? De la misma manera, el texto contenido en el alt es el que el lector de pantalla le anuncia al usuario y si tal atributo se omite, el lector de pantalla lee el nombre del archivo y si este no esta relacionado con el contenido de la imagen, esto genera una mala experiencia y ruido para el usuario. Y si se coloca vacío alt="" el lector de pantalla lo ignora porque lo interpreta como "no parte del contenido" y pasa de largo.

    Por lo tanto, su uso no es opcional para una imagen y podemos emplearlo de dos formas dependiendo el contexto:
    1. Descriptivo: Si la imagen es fundamental para el contenido porque este se auxilia de ella para ejemplificar o mostrar algo o la imagen es independiente, escribir un buen alt es fundamental en este caso.
    2. Vacío: Se coloca vacío si la imagen es decorativa, no ejemplifica nada visual ni textualmente, en este caso como mencionamos arriba, es ignorado por el lector de pantalla.
    Para escribir un buen alt nos podemos auxiliar de un criterio, haciendonos la pregunta de: Si por algún motivo la imagen no se carga o muestra ¿QUE TENDRÍA QUE SABER EL USUARIO PARA NO PERDER INFORMACIÓN?

    Además, un alt no es una descripción detalla y exhaustiva del contenido de la imagen, no se deben colocar detalles innecesarios que hagan que se pierda el sentido real de la imagen, con descripciones que involucren colores, calidad, marcos o medida de la imagen. En el caso de que un alt no alcance para describir todo el contenido de la imagen y se necesite hacer una versión extendida de él para las personas con discapacidad nos auxiliamos de ARIA , con sus atributos aria-describedby o aria-details y en el caso de que sea también de revelencia para usuarios visuales mediante un enlace a otra página que contenga la descripción.

    Otro caso especial es el uso de imágenes para botones y los logos de empresas o sitios.

    • Cuando una imagen corresponde al elemento button , el texto alternativo de la imagen debe corresponder a la función que ese símbolo realiza en lugar de colocar una descripción de ella como si fuera una imagen descriptiva.
      EJEMPLO:
                                      
                                    <button>
                                    <img src="lupa.jpeg" 
                                    alt="Buscar" />
                                    </button>
      n lugar de: 
                                      
                                    <button>
                                    <img src="lupa.jpeg"
                                    alt="Lupa para buscar contenido" />
                                    </button>
    • En el caso de los logos, solo se coloca el nombre o las siglas de la empresa o entidad a la que corresponde, omitiendo la palabra "logo" al principio de ella.
      EJEMPLO:
                                      
                                    <img src="logo_nationalgeographic.png"
                                    alt="National Geographic" />

    En conclusión, el elemento alt no debería ser opcional, o es descriptivo o se coloca vacío, si es descriptivo no se sobrecarga de detalles innecesarios y fuera de contexto pero no deberíamos omitirlo.

    Identificando un elemento SVG como una imagen

    Un elemento SVG (Scalable Vector Graphic) es una imagen, pero a diferencia de las imágenes de archivos JPG, JPEG, PNG, ect., está basado en vectores y no en píxeles y es interpretada directamente por el navegador por ser código.

    Aunque visualmente luzca como una imagen, no es tratada como tal por el navegador a menos que este contenida en un elemento <img> , de lo contrario para el navegador es una instrucción gráfica no una imagen en sí. Un elemento SVG puede reaccionar a eventos, animarse, contener texto, cambiar con CSS o JS y un elemento <img> no, solo se muestra y ya.

    Un SVG esta formado por elementos que pertenecen al lenguaje SVG no a HTML, los cuales describen formas.
